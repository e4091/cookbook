# Inclusive language

Mistral content reaches a global audience. All communication must be inclusive and respectful.

---

## Gender-neutral language

Use gender-neutral alternatives for common terms.

| Use this | Not this |
|---|---|
| chair, moderator | chairman |
| humanity, people, humankind | man, mankind |
| operates, staffs | mans |
| sales representative | salesman |
| synthetic, manufactured | manmade |
| workforce, staff, personnel | manpower |

Don't use *he*, *him*, *his*, *she*, *her*, or *hers* in generic references. Instead:

- Rewrite using the second person (*you*).
- Rewrite the sentence with a plural noun and pronoun.
- Use *the* or *a* instead of a pronoun (e.g., "the document").
- Refer to a person's role (*developer*, *user*, *reader*).
- Use *person* or *individual*.
- Use a plural pronoun (*they*, *their*, *them*) in generic references to a single person if you can't write around the problem. Don't use *he/she* or *s/he*.

| Use this | Not this |
|---|---|
| If you have the appropriate permissions, you can set other users' API keys. | If the user has the appropriate permissions, he can set other users' API keys. |
| Developers need access to the API in their environments, but they don't need production credentials. | A developer needs access to the API in his environment. |
| When the reader opens the notebook... | When the reader opens her notebook... |

When writing about a real person, use the pronouns that person prefers.

In fictitious scenarios, strive for diversity. Choose names that reflect a variety of gender identities and cultural backgrounds. Avoid stereotypes in job roles.

## Accessibility terms

In general, use people-first language: refer to the person first, then the disability.

In some cases, identity-first language is acceptable—some people and communities prefer it. Always defer to the preferences of the individual or community you're writing about.

Never use language with offensive or insensitive connotations.

| Preferred (people-first) | Acceptable (identity-first) | Do not use |
|---|---|---|
| Person who is blind, person with low vision | Blind person | Sight-impaired, vision-impaired |
| Person who is deaf, person with a hearing disability | Deaf person | Hearing-impaired |
| Person with limited mobility, person with a physical disability | Physically disabled person, wheelchair user | Crippled, lame, handicapped |
| Is unable to speak, uses sign language, uses synthetic speech | | Dumb, mute, non-verbal |
| Has multiple sclerosis, cerebral palsy, a seizure disorder, or muscular dystrophy | | Affected by, stricken with, suffers from, a victim of |
| Person without a disability | Non-disabled person, able-bodied person | Normal person, healthy person |
| Person with a prosthetic limb, person with a limb difference | Amputee | Maimed, missing a limb, birth defect |
| Person with a disability | Disabled person | The handicapped, people with handicaps |
| Person with cognitive, developmental, or learning disabilities | Learning disabled, dyslexic person | Slow learner, mentally handicapped, differently abled, special needs |
| Person with autism | Autistic person, neurodivergent person | Asperger's (as a label) |
| Functional needs | | Special needs |

Don't use words that imply pity: *stricken with*, *suffers from*, *victim of*. Don't mention a disability unless it's relevant.

## Racial and ethnic language

- Use title-style capitalization for: Asian, Black and African American, Hispanic and Latinx, Native American, Alaska Native, Native Hawaiian, Pacific Islander, and Indigenous Peoples.
- Lowercase *multiracial* and *white*.
- Don't make generalizations about people, countries, regions, or cultures.
- Don't use slang that could be considered cultural appropriation.

---

## Bias-free technical terminology

Avoid terms with racial or militaristic connotations.

| Use this | Not this |
|---|---|
| primary/subordinate | master/slave |
| allowlist / allow list | whitelist |
| denylist / block list | blacklist |
| perimeter network | demilitarized zone (DMZ) |
| stop responding | hang |

---

## Militaristic language

Avoid terms associated with violence and military action unless describing physical combat operations.

| Use this | Not this |
|---|---|
| address; protect against; respond to | combat; fight; eliminate |
| cyberattack chain | kill chain |
| threat actor, bad actor | attacker, adversary |
| multilayered approach | defense-in-depth (avoid in general content) |
| protect, safeguard | guard, ward |
| security; protection | frontline defense, fortifications |
| vulnerabilities; points of access | attack surface (in general content) |

**Never use** in technical documentation: *bomb*, *enemy lines*, *invade*, *nuke*, *strike*, *troops*, *air cover*.

It's OK to use *attack* when the type of attack is specified: "malware attack," "injection attack," "brute-force attack."

## Inclusive examples in code

- Use fictional but realistic names in code examples. Choose names from a variety of backgrounds (e.g., Amara, Kai, Sofia, Tariq—not just Anglo-Saxon names).
- Don't use examples that reflect only a Western or affluent lifestyle.
- Use neutral email addresses and placeholder content: `user@example.com`, not `john@example.com`.
- In API examples, use neutral placeholder values: `"your-api-key"`, not `"mykey123"`.

---

## Language about Mistral products

- Treat Mistral AI as a proper noun: always capitalize.
- Don't attribute human characteristics to models or APIs.
- Don't use phrases like "the AI wants to..." or "the model believes...".
- Use objective, descriptive language: "the model returns," "the API accepts," "the connector exposes."
